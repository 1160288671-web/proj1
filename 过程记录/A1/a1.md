### 1.3 输出数据模型

输出路径：`data/a1_data/hot_raw_{date}.json`（`{date}` 为采集日期，如 `2026-07-23`），每天独立文件。

```json
{
  "title": "热点标题",
  "heat_score": 1791285,
  "rank": 1,
  "url": "https://...",
  "domain_tag": "剧集",
  "source": "微博热搜",
  "source_id": "weibo_hot",
  "platform": "weibo",
  "list_type": "hot",
  "trend": "新",
  "fetch_time": "2026-07-22T14:30:00+08:00",
  "fingerprint": "81e12206"
}
```

新增字段说明：
- `list_type`：`"hot"`（热度榜）或 `"rising"`（增量榜）
- `trend`：趋势标记，有数据的源（微博 label_name、百度 hotChange）有值，其余为 `""`

输出信封：
```json
{
  "meta": { "schema_version": "0.3", "status": "ok", ... },
  "summary": { "total": 160, "hot": 120, "rising": 40 },
  "source_stats": { ... },
  "items": [ ... ]
}
```

**实测发现的关键结论：**
- **跨源重复普遍**："两个AI演员比内娱待爆艺人都火"同时出现在微博、百度、头条三个热榜（重复本身是爆款信号，保留）
- **领域标签大面积"综合"**：仅 B站热门视频的 `tname`（视频分区）能提供有效细分标签
- **热度值量级差极大**：头条千万级 vs B站热门播放量级（68万），A2 归一化至关重要
- **百度 `trend` 字段为文本标签**：`hotChange` 返回 "same"/"hot" 等而非变化数值，与预期不同
- **B站搜索趋势无热度值**：`heat_score` 全为 0，只能依靠排名顺序

## 二、A1 的核心原则

> A1 只做**原样结构化**，不做任何判断：
>
> - ❌ 不过滤（非娱乐内容、敏感内容都不滤，交给 A2）
> - ❌ 不归一化（热度值不跨平台归一化，交给 A2）
> - ❌ 不去重（跨源重复保留，因为是热度信号）
> - ❌ 不打分（不分高下，那是 A2 的事）

## 四、与 A2 的接口契约

A1 写入 `state["hot_raw"]` / `data/a1_data/hot_raw_{date}.json`，A2 读取后：

- 按 `list_type` 筛选：可对 `hot` 和 `rising` 条分别处理或统一处理
- 按 `domain_tag` 白名单过滤
- 按 `trend` 辅助判断：有"新/热/爆"标记的条目可以有加权考量
- 按 `heat_score` + `rank` 交叉参考热度
- 跨源重复条目不剔除（重复本身是信号）

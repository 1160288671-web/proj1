# 全自动棒读视频生成工作流 · 初版设计稿

> 版本：v0.1（开环初版，不含失败回路、合规审核、平台直发）
> 原则：LLM 做判断，代码做执行；agent 之间用 JSON 交接，agent 的前置知识用 .md skill 配置。

---

## 1. 总体结构

```
[热榜API/RSS] → A1 热点采集 → A2 选题裁决 → A3 素材采集清洗 → A4 编剧
                                                              ↓
[待发布目录] ← A6 发布打包 ← A5 音画制作 ←——————— (script.json)
                                  ↑
                        [免版权曲库 / TTS / ffmpeg 管线]

（预留）A7 数据回收：schema 本期定义，本体后实现
```

两条主线贯穿全程：

- **统一 ID**：`topic_id → material_id → script_id → video_id`，每份交接 JSON 必须携带上游 ID。
- **归因标签**：体验类型、情绪标签、选题类型（单一/组合）从 A2/A4 写入，逐级透传，供 A7 归因。

---

## 2. Agent 设计卡

### A1 热点采集（执行型）

| 项 | 内容 |
|---|---|
| 任务 | 从预设数据源拉取当日热榜，原样结构化，不做任何判断 |
| 数据源 | 平台官方热榜 API / 第三方榜单服务（数据源清单做成配置文件，agent 不自己找源） |
| 输入 | 数据源配置 |
| 输出 | `hot_raw.json`：榜单条目（标题、热度值、领域粗标签、来源、时间） |
| skill 要点 | 只写"调哪个接口、输出什么格式"，禁止 agent 临场写爬虫 |

### A2 选题裁决（判断型）

| 项 | 内容 |
|---|---|
| 任务 | 领域过滤（仅保留泛娱乐）→ 敏感过滤 → 单一/组合候选生成 → 四维打分 → 放行 3 个 |
| 组合逻辑 | 仅娱乐领域内两两组合；每个组合须先写出 1–3 条**关联假设**（见第 6 节），写不出合理假设的组合直接淘汰 |
| 打分维度 | 热度、安全性、可制作性、素材丰富度预判，各 1–5 分，加权求和（权重可配置） |
| 输入 | `hot_raw.json` |
| 输出 | `topics.json`（恰好 3 个选题，含关联假设与打分明细） |
| skill 要点 | 娱乐领域白名单、敏感词表、打分 rubric、组合假设的写法示例 |

### A3 素材采集清洗（执行型）

| 项 | 内容 |
|---|---|
| 任务 | query 生成 → 素材抓取 → 清洗去重 → 信息密度排序 |
| query 生成 | 单一选题：领域关键词直接扩展；组合选题：基于 A2 的关联假设转写为 3–5 个搜索词 |
| 数据源 | RSS / 开放新闻源 / 可合法抓取的文本源 |
| 清洗规则 | 见第 5 节（字数过滤、embedding 去重、信息密度打分均为代码执行，非 LLM 判断） |
| 输入 | `topics.json` |
| 输出 | `materials.json`（每个选题保留 Top N 素材，默认 N=8） |
| skill 要点 | 只定义输出 schema 和调用顺序；去重阈值、密度打分公式写在工具里 |

### A4 编剧（判断型）

| 项 | 内容 |
|---|---|
| 任务 | 读素材 → 定体验定位（主攻 1 + 辅攻 1，参照第 4 节映射表）→ 写口播稿 → 拆分镜 |
| 核心约束 | ① 情绪标签只能从受控词表（第 3 节）中选；② 分镜 schema 字段必须填全；③ 时长预算约束（见下） |
| 时长预算 | 默认目标 60–90 秒；中文 TTS 约 4–5 字/秒，口播稿总字数 = 目标时长 × 4.5，写稿前换算好 |
| 输入 | `topics.json` + `materials.json` |
| 输出 | `script.json`（机器交接）+ `脚本报告.md`（人读，含体验定位理由） |
| skill 要点 | 六体验映射表、情绪词表、分镜 schema、钩子写法（前 3 秒）、棒读语感要求（口语化、短句） |

### A5 音画制作（执行型，固定管线 + agent 调度）

| 子任务 | 执行方式 | 说明 |
|---|---|---|
| TTS 配音 | 工具 | 音色按情绪标签从受控音色表映射，语速默认 1.0 |
| BGM 选配 | 工具 + 标签匹配 | 免版权曲库按情绪标签检索，再按 BPM 与分镜节奏对齐（不做"乐理分析"） |
| 背景图 | 工具 | AI 生图或图库检索；全局统一风格后缀（写在 skill 里），保证同一条视频画风一致 |
| 字幕 | 工具 | 由 TTS 时间轴自动生成 |
| 合成渲染 | 固定脚本 | ffmpeg 管线，agent 只填参数不决策 |

- 输入：`script.json`
- 输出：成片 + `production.json`（记录了所有素材与参数，供溯源）
- skill 要点：音色表、曲库标签映射表、画面风格后缀、渲染参数模板

### A6 发布打包（执行型，人工放行）

| 项 | 内容 |
|---|---|
| 任务 | 成片 + 标题/简介/标签 + 封面，打包进待发布目录，生成 `publish.json`；不自动上传 |
| 输出 | 待发布目录 + `publish.json` |

### A7 数据回收（判断型，本期只定义接口）

- 输入：`video_id` + 平台后台数据（初期人工导出 CSV 喂入）
- 核心指标：完播率 > 5 秒留存 > 点赞率 > 评论率 > 分享率
- 输出：`performance.json`（指标 + 归因标签）+ `复盘报告.md`
- 回流动作：聚合归因标签，更新 A2 打分权重
- **本期依赖**：A2/A4 必须把体验类型、情绪标签、选题类型写进交接 JSON，否则后期无法归因

## 3. 接口 JSON 初稿库

### 3.0 通用信封（所有交接 JSON 的头部）

```json
{
  "meta": {
    "workflow_id": "wf-20260718-01",
    "stage": "A2_topic_select",
    "schema_version": "0.1",
    "created_at": "2026-07-18T10:00:00+08:00",
    "status": "ok"
  }
}
```

下游收到后先校验 `schema_version` 与必填字段，不通过即报错（开环版唯一的校验点，成本近零）。

### 3.1 topics.json（A2 → A3/A4）

```json
{
  "meta": {},
  "topics": [
    {
      "topic_id": "tp-001",
      "type": "combo",
      "domains": ["演唱会经济", "盲盒潮玩"],
      "title_candidate": "演唱会门票炒不动之后，年轻人开始炒什么",
      "keywords": ["演唱会", "黄牛", "潮玩", "盲盒"],
      "combo_hypotheses": [
        "演唱会受众与潮玩消费者高度重叠，均为 18-30 岁悦己消费人群",
        "演出票务溢价退潮后，替代性情绪消费品热度上升"
      ],
      "experience_hint": "优越感",
      "scores": { "heat": 4, "safety": 5, "feasibility": 4, "material_richness": 3, "total": 16 },
      "status": "approved"
    }
  ]
}
```

字段说明：`type` ∈ `single | combo`；`combo_hypotheses` 仅 combo 必填，写不出即淘汰；`experience_hint` 为建议体验定位，A4 可改但必须说明理由。

### 3.2 materials.json（A3 → A4）

```json
{

    }

}
```

### 3.3 script.json（A4 → A5）★ 全链路最关键接口

```json
{
  "meta": {},
  "script_id": "sc-001",
  "topic_id": "tp-001",
  "experience": { "primary": "优越感", "secondary": "爽感", "reason": "……" },
  "duration_target_sec": 75,
  "title_options": ["……", "……", "……"],
  "hook": "前3秒口播钩子",
  "full_text": "完整口播稿",
  "storyboard": [
    {
      "seq": 1,
      "duration_sec": 6,
      "voice_text": "该镜头配音文本",
      "subtitle_text": "字幕文本（可与配音略有出入）",
      "visual_prompt": "背景图提示词（系统统一追加风格后缀）",
      "emotion": "悬疑",
      "transition": "cut",
      "bpm_hint": "mid"
    }
  ]
}
```

硬性约束：`emotion` 取值必须 ∈ 受控词表；`transition` ∈ `cut | fade | slide`；`bpm_hint` ∈ `low | mid | high`；Σ `duration_sec` ≈ `duration_target_sec`。

## 4. 受控情绪词表（初稿）

编剧选标签、曲库检索、TTS 音色三方共用同一套词表。

| 标签 | 英文映射（曲库检索用） | 适用分镜段落 | BPM 参考 | TTS 语气建议 |
|---|---|---|---|---|
| 悬疑 | suspense / mystery | 开头钩子、抛问题 | 80–100 | 压低、放慢、留白 |
| 紧张 | tense / dramatic | 冲突升级段 | 110–130 | 语速加快、短句 |
| 激昂 | epic / uplifting | 高潮、观点输出 | 120–140 | 音量提升、重音加强 |
| 欢快 | upbeat / happy | 轻松信息段 | 100–120 | 明快、跳跃 |
| 戏谑 | quirky / playful | 吐槽、反讽段 | 95–115 | 拖腔、停顿梗 |
| 温暖 | warm / emotional | 人物故事段 | 70–90 | 柔和、亲切 |
| 治愈 | healing / gentle | 收尾、金句段 | 60–80 | 放缓、轻声 |
| 严肃 | serious / corporate | 数据、事实陈述段 | 90–105 | 平稳、克制 |

规则：
- 每个分镜只能选 1 个标签；一条视频内标签不超过 3 种（避免情绪碎片化）；
- 曲库标签向本词表做映射表，映射不到的一律归为"严肃"；
- TTS 音色同样按词表枚举（音色表初版可先设 2–3 个音色覆盖 8 种情绪）。

## 5. 体验映射表（初稿）

编剧的 skill 核心。每条脚本主攻 1 种、辅攻 1 种，选择结果与理由写入 `script.json`。

| 体验 | 适用选题特征 | 叙事结构 | 主攻情绪链 | 完播杠杆 |
|---|---|---|---|---|
| 掌控感 | 信息繁杂的热点事件 | 时间线梳理 / 清单体 | 严肃 → 治愈 | "一条视频讲清楚"的确定感 |
| 爽感 | 冲突、反转、打脸类事件 | 钩子 → 升级 → 反转 → 释放 | 紧张 → 激昂 | 前 3 秒直接抛冲突 |
| 共鸣感 | 普通人视角、群体性处境 | 个体故事 → 群体投射 | 温暖 → 治愈 | 弹幕/评论"是我了" |
| 优越感 | 信息差、冷知识、行业内幕 | "90%的人不知道" → 揭示 → 显摆点 | 悬疑 → 戏谑 | 社交货币（转发欲） |
| 成长感 | 方法论、可迁移的经验 | 问题 → 拆解 → 可执行清单 | 严肃 → 欢快 | 收藏率 |
| 参与感 | 争议性、两难话题 | 两难设定 → 各方观点 → 留白提问 | 戏谑 → 紧张 | 评论率 |

映射规则（写入 A2/A4 skill）：
- A2 裁决时按选题特征给 `experience_hint`；
- A4 可推翻 hint，但必须在 `reason` 字段说明；
- 主攻 + 辅攻不允许同质组合（如 爽感+参与感 可，掌控感+成长感 慎用——两者都偏理性，情绪曲线太平）。

## 6. 素材清洗与去重规则（替代原"平台优先级"规则）

全部由代码执行，不走 LLM：

1. **字数过滤**：正文字数 < 500 直接丢弃；
2. **相似度判重**：embedding 两两相似度 ≥ 0.85 判为重复簇；
3. **簇内保留规则**：信息密度打分（满分 9）——
   - 含具体数据/金额/百分比：+2
   - 含实名人物/机构：+2
   - 含明确时间线/事件脉络：+2
   - 含直接引语：+1
   - 含因果解释（非纯情绪宣泄）：+2
4. 同分保留发布更早者（原创概率高）；
5. 每选题输出 Top 8，不足 8 条则在 `materials.json` 里标记 `"material_warning": "insufficient"`，交由 A4 决定收窄选题或放弃（开环版允许 A4 直接跳过该选题并在报告中说明）。

---

## 7. 组合选题：关联假设生成流程（补入 A2/A3）

1. **A2 阶段**：对每个候选组合，LLM 生成 1–3 条关联假设（句式："A 的 X 与 B 的 Y 因为 Z 产生关联"）。写不出合格假设的组合直接淘汰——这本身就是可制作性打分的第一道筛子。
2. **A3 阶段**：将每条假设转写为 2–3 个搜索词（假设中的实体词 + 关系词），合并去重后执行检索。
3. **验证回填**：若某假设搜不到任何素材，在 `materials.json` 中标记该假设 `"verified": false`，A4 写脚本时应绕开未验证的假设。

## 8. 本期明确不做（边界声明）

- 失败回路与重试状态机（v2）
- 合规/版权/事实审核 agent（v2）
- 平台自动上传（本期仅打包待发布）
- 数据回收 agent 本体（本期仅定义 schema 与归因标签透传）
- history.jsonl 的自动排除逻辑（本期只追加记录）

## 9. 联调顺序建议

1. 先固定 1 个选题手动构造 `topics.json`，打通 A4 → A5 → A6（后半段是工程难点所在）；
2. 再接 A3，验证素材质量对脚本的影响；
3. 最后接 A1/A2 全自动跑一轮；
4. 跑通后再回头补 A7 与失败回路。
